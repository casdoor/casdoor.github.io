Общий обзор

Casdoor это UI-первая централизованная аутентификация / Single-Sign-On (SSO) платформа основанная на OAuth 2.0 / OIDC.

Casdoor обслуживает веб-интерфейс и запросы на регистрацию от пользователей приложения.

Характеристики корпуса:​

  1. Разработанная Golang отдельная архитектура для внешнего интерфейса, Casdoor поддерживает высокую совместимость, поддерживает веб-администрирование и поддерживает множественные языки (китайский, английский).

  2. Casdoor поддерживает вход с помощью плагинов на Github, Google, QQ, WeChat сторонних приложений.

  3. С Касбином на основе управления авторизацией, Casdoor поддерживает модели управления ACL, RBAC, ABAC, RESTful accessing контроля.

  4. Код подтверждения телефона, код подтверждения по электронной почте и забыли функции пароля.

  5. Доступ к прослушиванию журналов и записи.

  6. Облако Alibaba, облако Tencent Cloud, облачное хранилище изображений CDN облако.

  7. Настраиваемая регистрация, логин и забыли страницы с паролем.

  8. Casdoor поддерживает интеграцию с существующими системами с использованием метода синхронизации db, пользователи могут плавно переходить на Casdoor .

  9. Casdoor поддерживает основные базы данных: MySQL, PostgreSQL, SQL Server и т.д., и поддерживает расширение новой базы данных с плагинами.

Как это работает:​

Шаг 0 (Пре-знания)​

  1. Процесс авторизации Casdoor основан на OAuth 2. Поэтому настоятельно рекомендуется кратко взглянуть на то, как работает OAuth 2.0. введение OAuth 2.0.

Шаг 1 (запрос авторизации)​

Ваше приложение (может быть веб-сайтом или чем-то еще) должно создать URL в этом формате endpoint/login/oauth/authorize? lient_id=xxx&response_type=code&redirect_uri=xxx&scope=read&state=xxx. На URL заменить конечную точку URL-адресом вашего Casdoor'а и заменить xxx на вашу собственную информацию.

::note Подсказки

Как заполнить части xxx?

  • Для client_id: вы можете найти это в каждом отдельном приложении

  • Для redirect_uri: вы должны установить это на свой собственный URL обратного вызова приложения, с этой информацией, Кассдорф может знать, куда отправить информацию обратно после разрешения

  • Для состояния ``: вы должны заполнить это имя Заявления

:::

Приложение будет говорить с пользователем: "Привет, теперь мне нужны ресурсы и мне нужно ваше разрешение, чтобы взять эти ресурсы, Вы думаете перейти на этот URL-адрес и заполнить имя пользователя и пароль для меня?

С правильным составленным URL, ваше приложение сделает запрос на запуск пользователя на этот URL, и Запрос на авторизацию завершен.

Шаг 2 (Грант авторизации)​

Этот шаг прост: пользователь перенаправляется на URL, составленный выше, и пользователь должен видеть страницу входа из Casdoor.
Введя правильное имя пользователя и учетные данные в форме входа, Casdoor теперь знает личность пользователя, и собирается отправить два ключевых слова: код и состояние обратно на URL обратного вызова, установленный в Шаге 1.

Пользователь открывает URL и предоставляет учетные данные Casdoor. Касдорф будет говорить: "В поиске хорошего ~ это пользователь (который разрешает приложению получить код и состояние) я знаю в моей базе данных, и я вышлю код и состояние обратно в приложение, используя обратный URL (redirect_uri)"

С этими двумя ключевыми словами отправка обратно вашей заявке, теперь доступна для приложения, и таким образом Грант авторизации завершен.

tip

Касдорф также обеспечивает вход сторонних разработчиков, в этом случае вы не увидите регистрационные данные на странице ввода данных, а список сторонних провайдеров. Вы можете войти в ваше приложение, используя этих провайдеров, используя Кассдорф в качестве среднего слоя (средняя версия).

Шаг 3 (Грант авторизации)​

На этом шаге ваше приложение уже имеет код из Шага 2, и оно пойдет к Касдору: "Эй, теперь пользователь согласился дать мне код `, вы хотите проверить этот коди дать мнеaccess_token`?

Шаг 4 (Маркер доступа)​

In this step, Casdoor speaks back to the Application: "You know what, this code seems legit, you must be the right Application. Приходите, вот access_token.
с этим кодом, что Касдорф знает, что это авторизованная заявка (авторизация, предоставленная пользователем в Шаге 2) для получения access_token (будет использована позже, чтобы получить более полезные вещи).

Шаг 5 (Маркер доступа)​

На этом шаге ваше приложение сообщает: "Хороший, только что получил свежий и вкусный access_token, Теперь я могу использовать его, чтобы получить что-то более ценное с Ресурсного Сервера!

И ваше приложение переключается на ресурс: "Hey buddy, хотите проверить access_token out? Я получил это от Касдвери, и вы хотите посмотреть, верно ли это у вас с Касдор?"

Шаг 6 (защищенные ресурсы)​

Сервер ресурсов говорит с вашим приложением: "Не плохо ~ это похоже на то, что у меня с Casdoor, и Касдорф говорит, кто держит этот доступ_токен может иметь эти защищенные ресурсы. Теперь, ты сдаешься!"

И это в основном то, как работает Casdoor с вашей заявкой.

Подсказка

Касдорф может играть как Авторизация Сервера так и Ресурс Сервера , т.е. Касдорф уполномочивает нашу заявку на получение ресурсов (например, обычно текущую информацию о пользователе) из базы данных Casdoor.

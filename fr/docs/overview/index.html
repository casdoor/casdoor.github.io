Aperçu

Casdoor est une première plate-forme d'authentification centralisée / Single Sign-On (SSO) basée sur OAuth 2.0 / OIDC.

Casdoor sert à la fois l'interface web et les demandes de connexion des utilisateurs de l'application.

Caractéristiques de la porte :​

  1. Architecture séparée front-end et back-end, développée par Golang, Casdoor prend en charge la haute concurrence, fournit l'interface utilisateur basée sur le Web et prend en charge plusieurs langues (chinois, anglais).

  2. Casdoor prend en charge Github, Google, QQ, connexion aux applications tierces de WeChat, et supporte l'extension de connexion tierce avec des plugins.

  3. Avec la gestion d'autorisation basée sur Casbin , Casdoor prend en charge les modèles ACL, RBAC, ABAC, RESTful access control.

  4. Code de vérification du téléphone, code de vérification de l'e-mail et fonctions d'oubli du mot de passe.

  5. Accès à l'audit et à l'enregistrement des journaux.

  6. Alibaba Cloud, Tencent Cloud, Qiniu image cloud stockage CDN.

  7. Personnalisable les pages de registre, de connexion et d'oubli des mots de passe.

  8. Casdoor prend en charge l'intégration avec les systèmes existants en utilisant la méthode de synchronisation db, les utilisateurs peuvent passer à Casdoor en douceur.

  9. Casdoor prend en charge les bases de données principales : MySQL, PostgreSQL, SQL Server etc, et supporte l'extension de la nouvelle base de données avec des plugins.

Comment ça marche :​

Étape 0 (Pré-connaissance)​

  1. Le processus d'autorisation de Casdoor est basé sur l'OAuth 2. , donc il est fortement recommandé de jeter un coup d'oeil sur comment OAuth 2.0 fonctionne. Une introduction d'OAuth 2.0.

Étape 1 (demande d'autorisation)​

Votre application (peut être un site web ou quoi que ce soit) doit composer une URL dans ce format endpoint/login/oauth/authoriz? lient_id=xxx&response_type=code&redirect_uri=xxx&scope=read&state=xxx. Dans l'URL, remplacez endpoint par l'URL d'hôte de votre Casdoor, et remplacez xxx par vos propres informations.

Astuces

Comment remplir les pièces xxx?

  • Pour client_id: vous pouvez le trouver sous chaque application

  • Pour redirect_uri: vous devez définir ceci sur l'URL de rappel de votre propre application, avec cette information, Casdoor peut savoir où renvoyer des informations après autorisation

  • Pour l'état ``: vous devez remplir ceci avec le nom de votre Application

La Demande va parler à l'utilisateur : « Hey, maintenant j'ai besoin de quelques ressources et j'ai besoin de votre permission pour prendre ces ressources. Vous voulez aller à cette URL et remplir votre nom d'utilisateur et votre mot de passe pour moi?

Avec l'URL correctement composée, votre application va lancer une requête à cet URL, et la demande d'autorisation est terminée.

Étape 2 (Autorisation)​

Cette étape est simple : l'utilisateur est redirigé vers l'URL composée ci-dessus, et l'utilisateur doit voir la page de connexion de Casdoor.
En tapant le nom d'utilisateur et les identifiants corrects dans le formulaire de connexion, Casdoor connaît maintenant l'identité de l'utilisateur, et est sur le point d'envoyer deux mots clés : code et state back to the callback URL set in Step 1.

L'utilisateur ouvre l'URL et fournit les informations d'identification à Casdoor. Casdoor dira : "J'ai l'air bien ~ c'est l'utilisateur (qui autorise l'application à obtenir le code et `l'état`) que je connais dans ma base de données, et je renvoie le code et état à l'application en utilisant l'URL de rappel (redirect_uri)"

Avec ces deux mots-clés renvoyés à votre demande, l'authentification est maintenant accordée à l'application, et donc la Subvention d'Autorisation est terminée.

tip

Casdoor fournit également une connexion de tiers, dans ce cas, vous ne verrez pas les identifiants entrés dans la page mais une liste de fournisseurs tiers. Vous pouvez vous connecter à votre application en utilisant ces fournisseurs, avec Casdoor comme une couche intermédiaire (middleware).

Étape 3 (Autorisation)​

Dans cette étape, votre demande a déjà le code en main de l'étape 2, et il parlera à Casdoor: "Hey, maintenant l'utilisateur a accepté de me donner le code , vous voulez vérifier ce code et me donner le access_token?

Étape 4 (jeton d'accès)​

In this step, Casdoor speaks back to the Application: "You know what, this code seems legit, you must be the right Application. Venez, voici le access_token.
Avec ce code `, que Casdoor sait qu'il s'agit d'une application autorisée (autorisation donnée par le bon utilisateur à l'étape 2) en essayant d'obtenir le access_token` (sera utilisé plus tard pour obtenir des choses plus utiles).

Étape 5 (jeton d'accès)​

Dans cette étape, votre application dit : "Joli, vient de recevoir le nouveau access_tokenfraîchement savoureux, Je peux maintenant l'utiliser pour obtenir quelque chose de plus précieux du Serveur de Ressources!

Et votre Application se tourne vers le Serveur de Ressources: "Hey buddy, voulez-vous vérifier ce access_token dehors? Je l'ai obtenu de Casdoor et vous voulez voir si c'est la bonne que vous avez eu avec Casdoor?"

Étape 6 (Ressource Protégée)​

Le Serveur de Ressources revient à votre Application : "Pas mal ~ il semble juste que celui que j'ai eu avec Casdoor, et Casdoor dit qui détient ce access_token peut avoir ces ressources protégées. Maintenant, vous le prenez !"

Et c'est essentiellement ainsi que Casdoor fonctionne avec votre application.

Astuce

Casdoor peut jouer à la fois les parties Authorization Server et Resource Server i.e. Casdoor autorise notre Application à obtenir des ressources (par exemple les informations de l'utilisateur connecté actuellement) à partir de la base de données de Casdoor.

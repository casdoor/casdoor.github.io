概要

Casdoorは、OAuth 2.0/OIDCに基づく、UI初の集中認証/ シングルサインオン(SSO) プラットフォームです。

Casdoorは、Web UIとアプリケーションユーザーからのログイン要求の両方を提供します。

Casdoorの機能:​

  1. Golangによって開発されたフロントエンドとバックエンドの独立したアーキテクチャ、Casdoorは高い同時性をサポートし、Webベースの管理UIを提供し、複数の言語(中国語、英語)をサポートしています。

  2. Casdoorは、Github、Google、QQ、WeChatのサードパーティアプリケーションのログインをサポートし、プラグインによるサードパーティログインの拡張をサポートしています。

  3. Casdoorは、 Casbin ベースの認証管理により、ACL、RBAC、ABAC、RESTfulアクセス制御モデルをサポートしています。

  4. 電話確認コード、メール確認コード、パスワードの機能を忘れます。

  5. 監査や記録のログへのアクセス。

  6. Alibabaクラウド、テンセントクラウド、Qiniuクラウド画像CDNクラウドストレージ。

  7. カスタマイズ可能なレジスタ、ログイン、およびパスワードページを忘れます。

  8. Casdoorはdb同期方式を使用して既存のシステムとの統合をサポートし、ユーザーはCasdoorにスムーズに移行することができます。

  9. Casdoorは主流のデータベースをサポートしています:MySQL、PostgreSQL、SQL Serverなど、プラグインを使用して新しいデータベースの拡張をサポートしています。

仕組み：​

ステップ0 (事前知識)​

  1. Casdoorの認証プロセスは、OAuth 2に基づいて構築されます。 したがってプロトコルは、OAuth 2.0がどのように機能するかを簡単に確認することを強くお勧めします。 OAuth 2.0の 紹介。

ステップ1 (認証リクエスト)​

Your Application (could be a website or whatever) should compose an URL in this format endpoint/login/oauth/authorize?client_id=xxx&response_type=code&redirect_uri=xxx&scope=read&state=xxx. URLでは、 エンドポイント をCasdoorのホストURLに置き換え、 xxx を自分の情報に置き換えます。

ヒント

xxx パーツを記入するには？

  • client_id: それぞれのアプリケーションで見つけることができます

  • redirect_uri: 自分のアプリケーションのコールバックURLに設定する必要があります。 この情報でカスドアは承認後にどこに情報を送信するかを知ることができます

  • 状態: アプリケーション名を入力してください

The Application will speak to the user: "Hey, now I need some resources and I need your permission to take these resources, you mind go to this URL and fill out your username and password for me?"

正しく構成された URL では、アプリケーションはユーザーにこの URL へのリクエストを送信させ、 Authorization Request が行われます。

ステップ2（認証グラント）​

This step is straightforward: the user is redirected to the URL composed above, and the user should see the login page from Casdoor.
By typing the correct username and credential into the login form, Casdoor now knows the identity of the user, and is about to send two keywords: code and state back to the callback URL set in Step 1.

ユーザーはURLを開き、Casdoorに資格情報を提供します。 Casdoor will say: "Looking good ~ this is the user (who is authorizing the Application to get the code and state) I know in my database, and I will send the code and state back to the Application using the callback URL (redirect_uri)"

これら2つのキーワードがアプリケーションに送信されると、アプリにauthrizationが付与され、 Authorization Grant が完了します。

tip

Casdoorはまた、サードパーティのログインを提供します, この場合, あなたは、資格情報がページを入力するが、サードパーティのプロバイダのリストが表示されません. これらのプロバイダを使用してアプリにログインでき、Casdoorはミドルウェアとして使用できます。

ステップ3（認証グラント）​

In this step, your Application already has the code in hand from Step 2, and it will speak to Casdoor: "Hey, now the user agreed to give me the code, you wanna check this code out and give me the access_token?"

ステップ 4 (アクセストークン)​

このステップでは、Casdoorはアプリケーションに返信します。 「知っているか、この コード は合法に見えるが、正しいアプリケーションでなければならない。 Come, here's the access_token."
With this code, that Casdoor knows that it is an authorized Application (authorization given by the correct user in Step 2) trying to get the access_token (will be used later to get more useful things).

ステップ 5 (アクセストークン)​

In this step, your Application says: "Nice one, just got the fresh-and-tasty access_token, I can now use it to get something more valuable from the Resource Server!"

そして、あなたのアプリケーションは リソースサーバーに変わります: "ねえ、この access_token をチェックしますか？ 私はCasdoorからこれを手に入れました。あなたはこれがCasdoorと一緒に持っていた正しいものであるかどうかを確認したいですか？」

ステップ6（保護資源）​

The Resource Server speaks back to your Application: "Not bad ~ it seems just like the one I had with Casdoor, and Casdoor says whoever hold this access_token can have these Protected Resource. 今、あなたはそれを取る!"

これは基本的にCasdoorがアプリケーションと連携する方法です。

ヒント

Casdoor は、 Authorization Server と Resource Server の両方のパーツ、すなわちを再生できます。 Casdoorは、Casdoorのデータベースからリソース(通常はログインしているユーザー情報など)を取得することをアプリケーションに許可します。
